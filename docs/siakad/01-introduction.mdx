# Chapter 1: Introduction and Overview

## 1.1 What SIAKAD Is

SIAKAD (Sistem Informasi Akademik) is a web-based academic information system built for Universitas Riau's Sistem Informasi program. It manages the core workflows of a university department: student enrollment, course registration, grade management, attendance tracking, thesis supervision, and internship coordination.

The system serves four distinct user roles:

| Role | Description | Primary Functions |
|------|-------------|-------------------|
| **Superadmin** | System-wide administrator | Full CRUD on all entities, user management, master data |
| **Admin Fakultas** | Faculty-scoped administrator | Manages students, lecturers, classes, schedules within their faculty |
| **Dosen** | Lecturer | Grade entry, attendance management, thesis/internship supervision, academic advising |
| **Mahasiswa** | Student | Course registration (KRS), view grades/transcripts, attendance, thesis/internship tracking, AI advisor |

## 1.2 Business Modules

SIAKAD contains 18 functional modules:

### Administrative Modules
1. **Admin Dashboard** — Faculty-scoped statistics with Chart.js visualizations (student counts, grade distributions, enrollment trends)
2. **Mahasiswa Management** — CRUD operations for student records with CSV export capability
3. **Dosen Management** — CRUD operations for lecturer records with export
4. **User Management** — Superadmin-only polymorphic user creation (creates User + associated Mahasiswa or Dosen profile in a single transaction)
5. **Master Data** — Reference tables: Fakultas, Prodi, MataKuliah, Kelas, TahunAkademik, Ruangan

### Academic Core Modules
6. **KRS (Kartu Rencana Studi)** — Course registration with a draft → pending → approved/rejected lifecycle. Enforces SKS limits based on previous semester's IPS (GPA), checks class capacity, and prevents duplicate course enrollment.
7. **Penilaian (Grading)** — Bulk grade input by lecturers. Numeric scores convert to letter grades (A through E) with corresponding weight values using configurable conversion rules.
8. **Presensi (Attendance)** — Per-meeting attendance tracking with status categories: hadir (present), sakit (sick), izin (excused), alpa (absent). Calculates attendance percentages with a configurable minimum threshold (default 75%).
9. **KHS (Kartu Hasil Studi)** — Per-semester grade report showing courses taken, grades received, and IPS (semester GPA).
10. **Transkrip** — Cumulative academic record across all semesters with IPK (cumulative GPA) calculation.

### Supervision Modules
11. **Skripsi (Thesis)** — Full thesis lifecycle management from proposal through completion, with bimbingan (guidance session) tracking between student and supervisor.
12. **Kerja Praktek (Internship)** — Internship registration and progress tracking with logbook entries.
13. **Bimbingan (Academic Advising)** — Dosen PA (academic advisor) manages assigned students' KRS approvals.

### Supporting Modules
14. **LMS** — Learning management with materi (course materials) file upload/download and tugas (assignments) with student submissions.
15. **Kehadiran Dosen** — Lecturer check-in/checkout tracking for class attendance.
16. **AI Academic Advisor** — Gemini/Qwen API-powered chatbot that answers student questions grounded in their actual academic data. Includes a multi-layered guard system to prevent hallucination and off-topic responses.
17. **Notifications** — In-app notification system for schedule changes and academic events.
18. **Health Check** — System health monitoring endpoint checking database connectivity, cache availability, and storage access.

## 1.5 Architectural Decisions

Several architectural choices shape how the codebase works:

**What exists and functions:**
- **Controller → Service → Model** pattern for complex business domains (KRS, Penilaian, Presensi, Notification, Academic Calculation, AI Advisor)
- **Custom exception hierarchy** with `SiakadException` base class and `KrsException` subclass using static factory methods for each error condition
- **Faculty-scoped multi-tenancy** via middleware that filters data based on the logged-in admin's `fakultas_id`
- **Rate limiting** on sensitive endpoints: KRS operations (10/min), grading (20/min), AI chat (10/min)
- **Configurable academic rules** externalized to `config/academic_rules.php` and `config/siakad.php` (grade conversion tables, max SKS rules, KRS status flow)

**What was scaffolded but left incomplete:**
- **Repository pattern** — Three repository files exist (`KelasRepository`, `KrsRepository`, `NilaiRepository`) but contain no code. Services query Eloquent models directly.
- **DTO pattern** — Three DTO files exist (`NilaiDTO`, `KrsItemDTO`, `MahasiswaItemDTO`) but are empty. Data flows as Eloquent models and raw arrays.
- **Form Request validation** — Eight Form Request classes exist but all return `authorize() = false` and empty `rules()` arrays. Controllers perform inline validation instead.
- **Policy-based authorization** — Three policy files exist (`KelasPolicy`, `KrsPolicy`, `NilaiPolicy`) with no methods defined. Authorization is handled by `RoleMiddleware` and manual `if` checks in controllers.
- **Helper classes** — Three helper files exist (`FormatHelper`, `NilaiHelper`, `SemesterHelper`) but are empty. Utility logic lives inline in services.

**What does not exist:**
- No Enums, Events, Listeners, Jobs, Notifications (Laravel), or Mail classes
- No API endpoints (all routes return Blade views)
- No queue usage — all operations are synchronous
- No WebSocket or real-time features
- No file storage abstraction beyond local disk
