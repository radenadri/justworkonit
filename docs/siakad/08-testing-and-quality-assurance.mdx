# Chapter 8: Testing and Quality Assurance

## 8.1 Test Architecture Overview

SIAKAD uses **Pest v4** (built on top of PHPUnit v12) as its testing framework. The test suite contains **18 test files** with approximately **48 test cases** covering authentication, authorization, role-based page access, CRUD operations, and the AI academic advisor subsystem.

### Test Style Distribution

The codebase uses two distinct testing styles:

| Style | File Count | Description |
|-------|-----------|-------------|
| Pest closure-based | 17 files | `test('...', function () { })` — concise, expressive |
| PHPUnit class-based | 1 file | `class AcademicAdvisorTest extends TestCase` — used for complex setup |

All 17 Pest-style files rely on the global `Pest.php` configuration for their base class and traits. The single PHPUnit-style file (`AcademicAdvisorTest.php`) declares `use RefreshDatabase` directly within the class.

## 8.2 Test Configuration

### Pest.php — Global Setup

```php
// tests/Pest.php
pest()->extend(Tests\TestCase::class)
    ->use(Illuminate\Foundation\Testing\RefreshDatabase::class)
    ->in('Feature');

expect()->extend('toBeOne', function () {
    return $this->toBe(1);
});

function something()
{
    // placeholder for shared test helpers
}
```

Three things happen here:

1. **Base class binding**: All test closures in `Feature/` are bound to `Tests\TestCase`, which extends `Illuminate\Foundation\Testing\TestCase`. This provides `$this->get()`, `$this->post()`, `$this->actingAs()`, and all other HTTP testing methods.

2. **RefreshDatabase**: Applied automatically to every Feature test. Each test runs inside a database transaction that rolls back after completion, giving every test a clean database state.

3. **Custom expectation**: `toBeOne()` is registered as a custom Pest expectation (currently unused in the test suite).

### TestCase.php — Base Class

```php
// tests/TestCase.php
namespace Tests;

use Illuminate\Foundation\Testing\TestCase as BaseTestCase;

abstract class TestCase extends BaseTestCase
{
    //
}
```

The base test case is empty — no global `setUp()`, no shared fixtures, no custom assertions. All setup logic lives in individual test files via `beforeEach()` blocks or PHPUnit's `setUp()` method.

### Database Strategy

Feature tests use `RefreshDatabase`, which wraps each test in a database transaction. The test database is configured via `phpunit.xml` — typically SQLite in-memory for speed or a dedicated MySQL/PostgreSQL test database.

Unit tests do **not** get `RefreshDatabase` applied (the `->in('Feature')` scope excludes `Unit/`). The single unit test (`ExampleTest.php`) performs no database operations.

### 8.3.3 Authentication Tests

#### `tests/Feature/Auth/EmailVerificationTest.php` — Email Verification

**Purpose**: Tests the email verification screen, successful verification via signed URL, and rejection of invalid hashes.

**Key pattern**: Uses `Event::fake()` to capture dispatched events and `URL::temporarySignedRoute()` to generate valid/invalid verification URLs.

```php
test('email can be verified', function () {
    $user = User::factory()->unverified()->create();
    Event::fake();

    $verificationUrl = URL::temporarySignedRoute(
        'verification.verify',
        now()->addMinutes(60),
        ['id' => $user->id, 'hash' => sha1($user->email)]
    );

    $response = $this->actingAs($user)->get($verificationUrl);

    Event::assertDispatched(Verified::class);
    expect($user->fresh()->hasVerifiedEmail())->toBeTrue();
});
```

#### `tests/Feature/Auth/PasswordResetTest.php` — Password Reset Request

**Purpose**: Validates the "forgot password" screen renders and that reset links can be requested. Uses `Notification::fake()` to assert a password reset notification is sent.

#### `tests/Feature/Auth/PasswordUpdateTest.php` — Password Change

**Purpose**: Verifies authenticated users can change their password with the correct current password and are rejected with the wrong one.

#### `tests/Feature/Auth/PasswordConfirmationTest.php` — Password Confirmation Gate

**Purpose**: Tests the password confirmation screen (used before sensitive actions) renders and accepts valid credentials.

```php
});
```

## 8.5 Test Patterns

### Pattern 1: Factory-Based User Creation

All tests create users through `User::factory()->create()` with role overrides:

```php
// Default user (no role specified)
$user = User::factory()->create();

// Role-specific users
$admin = User::factory()->create(['role' => 'admin']);
$mahasiswa = User::factory()->create(['role' => 'mahasiswa']);
$dosen = User::factory()->create(['role' => 'dosen']);
```

When a role requires a linked profile record (Mahasiswa or Dosen), the test explicitly creates it:

```php
$mahasiswa = Mahasiswa::factory()->create([
    'user_id' => $user->id,
    'prodi_id' => $prodi->id,
]);
```

### Pattern 2: `actingAs()` for Authenticated Requests

Every authenticated request uses Laravel's `actingAs()` to set the session user:

```php
$response = $this->actingAs($user)->get(route('mahasiswa.dashboard'));
```

This avoids posting login credentials in every test and keeps tests focused on the feature being tested.

### Pattern 3: `beforeEach()` for Shared Fixtures

Tests that share a common data setup use Pest's `beforeEach()` to create fixtures once per test:

```php
beforeEach(function () {
    TahunAkademik::factory()->create(['is_active' => true]);
    $this->user = User::factory()->create(['role' => 'mahasiswa']);
    $this->prodi = Prodi::factory()->create();
    $this->mahasiswa = Mahasiswa::factory()->create([
        'user_id' => $this->user->id,
        'prodi_id' => $this->prodi->id,
    ]);
});
```

Because `RefreshDatabase` rolls back after each test, the `beforeEach` block runs fresh for every test — no data leaks between tests.

### Pattern 4: Cross-Role Denial Testing

The test suite verifies role boundaries by having one role attempt to access another role's routes:

```
  ┌──────────────┐     GET     ┌─────────────────────┐
  │  Mahasiswa    │────────────▶│ admin.fakultas.index │──── 403
  └──────────────┘             └─────────────────────┘

  ┌──────────────┐     GET     ┌─────────────────────┐
  │  Dosen        │────────────▶│ mahasiswa.dashboard  │──── 403
  └──────────────┘             └─────────────────────┘

  ┌──────────────┐     GET     ┌─────────────────────┐
  │  Mahasiswa    │────────────▶│ dosen.penilaian     │──── 403
  └──────────────┘             └─────────────────────┘

  ┌──────────────┐     GET     ┌─────────────────────┐
  │  Dosen        │────────────▶│ admin.dashboard     │──── 403
  └──────────────┘             └─────────────────────┘
```

This pattern catches middleware misconfigurations that would expose one role's data to another.

### Pattern 5: Direct Model Creation (No Factory)

The `AcademicAdvisorTest` deliberately avoids factories and creates models with `Model::create()`:

```php
$fakultas = Fakultas::create(['nama' => 'Fakultas Teknik']);
$prodi = Prodi::create([
    'fakultas_id' => $fakultas->id,
    'nama' => 'Sistem Informasi',
]);
```

This approach is used when the test needs precise control over field values (e.g., specific NIM, specific course codes, exact SKS values) that factory defaults would not provide.

### Pattern 6: Event and Facade Faking

The email verification test uses Laravel's fake infrastructure to capture side effects:

```php
Event::fake();
// ... trigger verification ...
Event::assertDispatched(Verified::class);
```

The password reset test uses `Notification::fake()` similarly. This prevents real emails from being sent during tests while still verifying the correct events/notifications fire.

## 8.6 Case Study: AcademicAdvisorTest

The `AcademicAdvisorTest` is the most complex test file in the suite. It is the only file that uses PHPUnit class-based style rather than Pest closures. It tests the AI Academic Advisor subsystem: context building, guard validation, and the full advisory pipeline.

### Why PHPUnit Style?

The test requires:
- Complex `setUp()` logic that creates 29+ database records (courses across multiple semesters)
- Class-level properties (`$contextBuilder`, `$guards`, `$mahasiswa`, `$activeTahunAkademik`, `$dosen`)
- A helper method `createCompletedCourses()` that loops to generate courses up to a target SKS count

These patterns map more naturally to a PHPUnit class than to Pest closures with `beforeEach()`.

### Test Data Setup

The `setUp()` method constructs a complete academic dataset:

```
setupTestData() creates:
├── Fakultas: "Fakultas Teknik"
├── Prodi: "Sistem Informasi"
├── User (dosen) → Dosen (NIDN: 1234567890)
├── User (mahasiswa) → Mahasiswa (NIM: 2303113649, angkatan: 2023)
├── TahunAkademik (2024/2025 Ganjil, active)
└── createCompletedCourses(87):
    ├── 29 MataKuliah records (3 SKS each, 87 total)
    ├── 29 Kelas records
    ├── Multiple TahunAkademik records (inactive, past semesters)
    ├── 29 Krs + KrsDetail records (status: approved)
    └── 29 Nilai records (nilai_angka: 85, nilai_huruf: A, status: final)
```

### Test Breakdown

#### Test 1: SKS Progress Calculation

Verifies the `AdvisorContextBuilder` correctly calculates graduation progress against the 144-SKS target.

```php
public function test_it_returns_total_sks_and_progress_using_144_rule(): void
{
    $context = $this->contextBuilder->build($this->mahasiswa);
    $progress = $this->contextBuilder->calculateGraduationProgress($context);

    $this->assertEquals(87, $progress['sks_lulus']);
    $this->assertEquals(144, $progress['sks_target']);
    $this->assertEquals(57, $progress['sks_remaining']);
    $this->assertEqualsWithDelta(60.4, $progress['progress_percent'], 0.5);
    $this->assertEquals(144, $context['prodi_rules']['graduation_total_sks']);
}
```

This test validates arithmetic correctness: 87/144 = 60.4%, 144 - 87 = 57 remaining.

#### Test 2: Thesis Eligibility

With 87 SKS completed (below the 144 threshold), the student should not be eligible for thesis:

```php
$this->assertFalse($progress['eligible_thesis']);
$this->assertEquals(57, $progress['sks_remaining']);
$this->assertEquals(144, $context['prodi_rules']['thesis_min_sks']);
```

#### Test 3: Course Status Detection

Verifies the context builder correctly categorizes courses from the Prodi curriculum config into three statuses:

| Status | Meaning | Source |
|--------|---------|--------|
| `LULUS` | Passed | Has a `Nilai` record with passing grade |
| `SEDANG_DIAMBIL` | Currently enrolled | In active KRS, no grade yet |
| `TERSEDIA_DI_KURIKULUM` | Available but not taken | In curriculum config, not in any KRS |

The test creates an enrolled course (no grade) and checks that it appears as `SEDANG_DIAMBIL`, while a curriculum course like "Big Data" appears as `TERSEDIA_DI_KURIKULUM`.

#### Test 4: Attendance Guard

When no attendance data exists (no `Pertemuan` records), the guard should:
- Set `data_available` to `false`
- Flag `all_zero_or_null` as `true`
- Reject LLM output that claims "attendance is low" (since there is no data to support that claim)

```php
$badOutput = 'Berdasarkan data, presensi rendah dan perlu diperbaiki.';
$guardResult = $this->guards->attendanceGuard($context, $badOutput);

$this->assertEquals(AdvisorGuards::GUARD_FAIL, $guardResult['status']);
$this->assertStringContainsString('belum tersedia', $guardResult['recommended_response']);
```

#### Test 5: Generic Assumption Rejection

The advisor must not use vague phrases like "biasanya" (usually), "umumnya" (generally), or "tergantung" (depends). The guard catches these:

```php
$forbiddenOutputs = [
    'Biasanya SKS kelulusan adalah 120 SKS.',
    'Umumnya mahasiswa semester 5 sudah bisa mengambil skripsi.',
    'Hal ini tergantung kebijakan prodi masing-masing.',
];

foreach ($forbiddenOutputs as $output) {
    $result = $this->guards->preventGenericAssumptions($output);
    $this->assertEquals(AdvisorGuards::GUARD_RETRY, $result['status']);
}
```

When a forbidden phrase is detected, the guard returns `GUARD_RETRY` with a retry prompt and sanitized output for the LLM to regenerate.

#### Test 6: Course Status Differentiation

A comprehensive test that creates all three course states in a single test and verifies the context builder distinguishes between them:

```php
$completedCourses = $statuses->where('status', 'LULUS');

$enrolledCourse = $statuses->firstWhere('kode', 'ENROLLED01');

$this->assertGreaterThan(0, $availableCourses->count());
```

#### Test 7: Guard Validation Rules

Verifies that `assertRulesPresent()` throws `InvalidArgumentException` when the context contains invalid prodi rules (e.g., `graduation_total_sks` = 0).

#### Test 8: Full Guard Pipeline

Runs the complete post-processing guard pipeline on both valid and invalid LLM output:

## 8.7 Code Quality Tools

### Laravel Pint

SIAKAD uses **Laravel Pint v1** for automated code formatting. Pint enforces a PSR-12-based style with Laravel-specific conventions.

**Command**:
```bash
vendor/bin/pint --dirty --format agent
```

The `--dirty` flag formats only files with uncommitted changes (staged or unstaged in git). The `--format agent` flag outputs machine-readable results.

Pint is the **only** code quality tool configured. It handles:
- Indentation and spacing
- Import ordering
- Brace placement
- Trailing commas
- Type declaration formatting

### What Is Missing

| Tool | Purpose | Status |
|------|---------|--------|
| Laravel Pint | Code formatting (PSR-12) | **Configured** |
| PHPStan / Larastan | Static type analysis | **Not configured** |
| PHP CS Fixer | Additional code style rules | Not needed (Pint covers this) |

| Code coverage reporting | Line/branch coverage metrics | **Not configured** |

The absence of PHPStan/Larastan means type errors in method signatures, return types, and property access are caught only at runtime. No CI/CD pipeline means tests must be run manually before deployment.

**1a. KRS Operations**

Write tests for the full KRS lifecycle:

Set an initial threshold of 60% and increase it as tests are added.
