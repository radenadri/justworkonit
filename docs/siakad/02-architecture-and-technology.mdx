## 2.2 Design Patterns

### Controller → Service → Model

The application uses a layered architecture for complex business domains. Controllers handle HTTP concerns (request parsing, response rendering), delegate business logic to service classes, and services interact with Eloquent models directly.

```
HTTP Request
    │
    ▼
Controller (validation, auth checks)
    │
    ▼
Service (business logic, transactions, calculations)
    │
    ▼
Eloquent Model (database queries, relationships)
    │
    ▼
Database
```

**Services that follow this pattern:**

| Service | Responsibility | Injected Into |
|---------|---------------|---------------|
| `KrsService` | Course registration lifecycle | `Mahasiswa\KrsController` |
| `PenilaianService` | Grade entry and conversion | `Dosen\PenilaianController` |
| `PresensiService` | Attendance recording and reporting | `Dosen\PresensiController`, `Mahasiswa\PresensiController` |
| `AkademikCalculationService` | IPS/IPK calculation, transcripts | Multiple controllers |
| `AkademikService` | Master data CRUD with caching | Admin controllers |
| `NotificationService` | Notification dispatch | `NotificationController` |
| `UserService` | Polymorphic user creation | `Admin\UserController` |
| `JadwalService` | Schedule conflict detection | Jadwal-related controllers |
| `AiAdvisorService` | AI chat orchestration | `Mahasiswa\AiAdvisorController` |
| `CacheService` | Centralized cache management | `WarmCacheCommand` |
| `ActivityLogService` | Audit trail recording | Various (via DI) |

**Controllers that bypass services and query models directly:** Most Admin CRUD controllers (MahasiswaController, DosenController, KelasController, etc.) perform Eloquent operations inline without a service layer.

### Exception Hierarchy

```
SiakadException (abstract)
    │
    ├── render(Request): Response
    │   ├── JSON response if expectsJson()
    │   └── Redirect with error flash message otherwise
    │
    └── KrsException (concrete)
        ├── noActiveSemester()     → code 404
        ├── alreadySubmitted()     → code 403
        ├── locked()               → code 403
        ├── classFull()            → code 422
        ├── courseAlreadyTaken()    → code 422
        ├── sksLimitExceeded()     → code 422
        ├── emptyKrs()             → code 422
        └── invalidStatus()        → code 422
```

Each static factory method returns a preconfigured exception instance with a human-readable Indonesian message, HTTP status code, and contextual data array. The `render()` method on `SiakadException` handles both API and web response formats.

## 2.4 Middleware Stack

Middleware is registered in `bootstrap/app.php`:

### Global Middleware
Standard Laravel middleware stack (encryption, session, CSRF, etc.) — no custom global middleware.

### Route-Level Middleware

| Alias | Class | Purpose |
|-------|-------|---------|
| `role` | `App\Http\Middleware\RoleMiddleware` | Checks `Auth::user()->role` against a comma-separated list of allowed roles. Returns 403 if unauthorized. |
| `fakultas.scope` | `App\Http\Middleware\FakultasScopeMiddleware` | Reads `Auth::user()->fakultas_id` and shares it as `$currentFakultasId` with all views. Applied to admin routes. |

### Rate Limiters (defined in `bootstrap/app.php`)

| Limiter Name | Rate | Applied To |
|-------------|------|-----------|
| `krs` | 10 requests/minute per user | KRS add/remove/submit endpoints |
| `penilaian` | 20 requests/minute per user | Grade input endpoints |
| `sensitive` | 30 requests/minute per user | User creation, password changes |
| `ai-chat` | 10 requests/minute per user | AI advisor chat endpoint |

### Middleware Application Pattern

```php
// routes/admin/*.php
Route::middleware(['auth', 'role:superadmin,admin_fakultas', 'fakultas.scope'])
    ->prefix('admin')
    ->name('admin.')
    ->group(function () {
        // Admin routes
    });

// routes/dosen/*.php
Route::middleware(['auth', 'role:dosen'])
    ->prefix('dosen')
    ->name('dosen.')
    ->group(function () {
        // Dosen routes
    });

// routes/mahasiswa/*.php
Route::middleware(['auth', 'role:mahasiswa'])
    ->prefix('mahasiswa')
    ->name('mahasiswa.')
    ->group(function () {
        // Mahasiswa routes
    });
```

## 2.5 Request Lifecycle

A typical request flows through these layers:

```
Browser Request
    │
    ▼
Laravel HTTP Kernel (global middleware)
    │
    ▼
Route Matching (routes/web.php includes role-prefixed route files)
    │
    ▼
Route Middleware ['auth', 'role:X', 'fakultas.scope']
    │  ├── auth: verify session, redirect to /login if guest
    │  ├── role: check user->role against allowed list, 403 if denied
    │  └── fakultas.scope: share fakultas_id with views (admin only)
    │
    ▼
Rate Limiter (if applied to route)
    │
    ▼
Controller Method
    │  ├── Inline validation (or occasionally Form Request)
    │  ├── Service call (for complex logic)
    │  │   └── Eloquent queries, DB transactions
    │  └── Return view() with data
    │
    ▼
Blade Template Rendering
    │  ├── Layout: x-app-layout (sidebar + header)
    │  ├── Data binding via {{ $variable }}
    │  └── Alpine.js x-data for interactivity
    │
    ▼
HTML Response to Browser
```

## 2.6 Frontend Architecture

### Build Pipeline

```
resources/js/app.js ──┐
resources/css/app.css ─┤──► Vite 7 (via Bun) ──► public/build/
                       │
@vite(['resources/css/app.css', 'resources/js/app.js'])
```

`app.js` imports Alpine.js and starts it. `bootstrap.js` configures Axios with default headers. No other JavaScript modules exist — all page-specific JS is written inline in Blade templates via `@push('scripts')`.

### Alpine.js Usage Patterns

Alpine.js handles all client-side interactivity:

| Pattern | Where Used | Implementation |
|---------|-----------|----------------|
| Sidebar collapse | Layout body | `x-data="{ sidebarOpen }"` with `$watch` + localStorage |
| Dropdown toggle | Header, navigation | `x-data="{ open: false }"` with `@click.outside` |
| Modal dialog | Admin CRUD pages | Vanilla JS `openModal()/closeModal()` toggling `hidden` class |
| Accordion | KRS available classes | `x-data="{ open: false }"` with `x-show` + `x-collapse` |
| Tab selection | Jadwal page | `x-data="{ selectedDay: '...', viewMode: 'card' }"` |
| Full chat app | AI Advisor | `x-data="aiAdvisor()"` with async fetch, message formatting |
| Dark mode | Layout header | Toggle with localStorage + class strategy |

### Chart.js Integration

Chart.js is loaded via CDN on dashboard pages. Data is passed from PHP to JavaScript using Blade's `@json()` directive within `@push('scripts')` blocks. Charts are dark-mode-aware, adjusting colors based on the active theme.

## 2.8 AI Integration Architecture

The AI Academic Advisor module has its own layered architecture:

```
AiAdvisorController
    │
    ▼
AiAdvisorService (orchestrator)
    ├── AdvisorContextBuilder (data gathering)
    │   ├── AkademikCalculationService (GPA, transcript)
    │   └── PresensiService (attendance data)
    ├── AdvisorGuards (input/output validation)
    │   ├── Pre-guards: assertRulesPresent, validateContext
    │   └── Post-guards: outOfContext, assumptions,
    │       attendance, courseReference, numericValidation
    └── LLM API call (Qwen or Gemini)
        ├── System prompt from resources/prompts/
        ├── Student context injected as JSON
        └── Retry mechanism if guards fail
```

The guard system prevents the AI from:
- Responding to non-academic questions (out-of-context guard)
- Using assumption language like "biasanya" or "umumnya" (generic assumption guard)
- Claiming attendance is low when no attendance data exists (attendance guard)
- Referencing course codes that do not exist in the student's curriculum (course reference guard)
- Stating incorrect numeric values for SKS or GPA (numeric validation guard)

If a post-guard fails with `GUARD_RETRY`, the system re-calls the LLM with the original conversation plus a correction prompt. Maximum retry count is 1.
