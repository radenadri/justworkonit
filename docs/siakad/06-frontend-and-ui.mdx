# Chapter 6: Frontend and UI

## 6.1 Frontend Architecture Overview

SIAKAD uses a server-rendered frontend built on Laravel Blade templates. There is no single-page application framework. Every page request returns fully rendered HTML from the server. Client-side interactivity is handled by Alpine.js for reactive UI components and vanilla JavaScript for DOM manipulation.

The rendering pipeline follows this path:

```
┌──────────────────────────────────────────────────────────────────────┐
│                        REQUEST LIFECYCLE                             │
│                                                                      │
│  Browser Request                                                     │
│       │                                                              │
│       ▼                                                              │
│  Laravel Router ──► Controller ──► Returns view('page', $data)       │
│                                        │                             │
│                                        ▼                             │
│                              Blade Template Engine                   │
│                                        │                             │
│                         ┌──────────────┼──────────────┐              │
│                         │              │              │              │
│                         ▼              ▼              ▼              │
│                    Layout          Components     Partials            │
│                 (app.blade)      (x-modal, etc)  (@include)          │
│                         │              │              │              │
│                         └──────────────┼──────────────┘              │
│                                        │                             │
│                                        ▼                             │
│                              Compiled HTML Response                  │
│                                        │                             │
│                                        ▼                             │
│                              Browser Renders Page                    │
│                                        │                             │
│                         ┌──────────────┼──────────────┐              │
│                         │              │              │              │
│                         ▼              ▼              ▼              │
│                    Alpine.js      Vanilla JS      Chart.js           │
│                  (reactivity)   (DOM, AJAX)    (dashboards)          │
│                                                                      │
│  Assets served from public/build/ (compiled by Vite + Bun)          │
│  Chart.js loaded via CDN (not bundled)                               │
└──────────────────────────────────────────────────────────────────────┘
```

### Technology Stack Summary

| Technology | Version | Role | Loading Method |
|---|---|---|---|
| Laravel Blade | — | Server-side HTML rendering | Built into Laravel |
| Alpine.js | v3 | Reactive UI interactions | Bundled via Vite |

| Vite | 7 | Asset bundling + HMR | Runs via Bun |
| Chart.js | latest | Dashboard data visualization | CDN (`jsdelivr.net`) |

#### Sidebar State Management

The sidebar collapse state is persisted across page loads using `localStorage`:

```html
<div x-data="{
    sidebarOpen: localStorage.getItem('sidebarOpen') !== 'false'
}" x-init="$watch('sidebarOpen', val => localStorage.setItem('sidebarOpen', val))">
    <!-- Sidebar toggle button -->
    <button @click="sidebarOpen = !sidebarOpen">
        <!-- hamburger icon -->
    </button>

    <!-- Sidebar panel -->
    <aside x-show="sidebarOpen" x-transition>
        <!-- navigation links -->
    </aside>
</div>
```

When `sidebarOpen` is `false`, the sidebar collapses and the main content area expands to full width. The `$watch` on Alpine.js writes the new state to `localStorage` on every toggle, so refreshing the page retains the user's preference.

#### Role-Based Navigation

The sidebar renders different menu items based on the authenticated user's `role` attribute. There are three distinct navigation sets:

| Role | Menu Items |
|------|-----------|
| `superadmin` / `admin` | Dashboard, Mahasiswa, Dosen, Users, Fakultas, Prodi, Mata Kuliah, Kelas, Jadwal, Tahun Akademik, Ruangan, LMS, Skripsi, Kerja Praktek |
| `dosen` | Dashboard, Jadwal, Presensi, Penilaian, Bimbingan, Kehadiran, LMS, Skripsi, Kerja Praktek |
| `mahasiswa` | Dashboard, KRS, KHS, Transkrip, Jadwal, Presensi, LMS, Skripsi, Kerja Praktek, AI Advisor |

The implementation uses Blade's `@if` / `@elseif` directives checking `auth()->user()->role`:

```blade
@if(auth()->user()->role === 'superadmin' || auth()->user()->role === 'admin')
    <a href="{{ route('admin.mahasiswa.index') }}">Mahasiswa</a>
    <a href="{{ route('admin.dosen.index') }}">Dosen</a>
    {{-- ... --}}
@elseif(auth()->user()->role === 'dosen')
    <a href="{{ route('dosen.jadwal.index') }}">Jadwal</a>
    {{-- ... --}}
@elseif(auth()->user()->role === 'mahasiswa')
    <a href="{{ route('mahasiswa.krs.index') }}">KRS</a>
    {{-- ... --}}
@endif
```

Note: `@can` / `@cannot` policy-based directives are **not** used. Authorization checks are based on the `role` string field directly.

#### Header Bar Components

The top header bar contains four elements rendered left-to-right:

1. **Breadcrumbs** — Shows the current page path (e.g., Dashboard > Mahasiswa > Create)
2. **Dark mode toggle** — Button that toggles `.dark` class on `<html>` and persists to `localStorage`
3. **Notification dropdown** — Loaded via `@include('components.notification-dropdown')`, shows recent notifications
4. **User avatar dropdown** — Displays user name and avatar with links to profile and logout

#### Mobile Bottom Navigation

Visible only on small screens (`sm:hidden`) and only for `mahasiswa` role. Contains five fixed icons at the bottom of the viewport:

```
┌──────────────────────────────────────────┐
│  🏠        📋        📅        ✅        ≡  │
│ Dashboard   KRS     Jadwal   Presensi  Menu │
└──────────────────────────────────────────┘
```

This bar replaces the sidebar on mobile. Dosen and admin users do not get a bottom navigation bar — they use a hamburger-triggered sidebar overlay on mobile instead.

## 6.5 Alpine.js Interaction Patterns

Alpine.js v3 provides all client-side reactivity. It is imported and initialized in `resources/js/app.js` with no custom plugins.

### 6.5.1 Sidebar Toggle

```html
<div x-data="{
    sidebarOpen: localStorage.getItem('sidebarOpen') !== 'false'
}" x-init="$watch('sidebarOpen', val => localStorage.setItem('sidebarOpen', val))">

    <button @click="sidebarOpen = !sidebarOpen" class="p-2">
        <svg x-show="!sidebarOpen"><!-- hamburger icon --></svg>
        <svg x-show="sidebarOpen"><!-- close icon --></svg>
    </button>

    <aside :class="sidebarOpen ? 'w-64' : 'w-0 overflow-hidden'"
           class="transition-all duration-300">
        <!-- navigation items -->
    </aside>
</div>
```

State is read from `localStorage` on page load and written back on every change. This means the sidebar stays collapsed or expanded across full page navigations.

### 6.5.2 Dropdown Menus

```html
<div x-data="{ open: false }" class="relative">
    <button @click="open = !open">
        Options
    </button>

    <div x-show="open"
         @click.away="open = false"
         x-transition:enter="transition ease-out duration-200"
         x-transition:enter-start="opacity-0 scale-95"
         x-transition:enter-end="opacity-100 scale-100"
         x-transition:leave="transition ease-in duration-75"
         x-transition:leave-start="opacity-100 scale-100"
         x-transition:leave-end="opacity-0 scale-95"
         class="absolute right-0 mt-2 w-48 bg-white rounded-md shadow-lg z-50">
        <!-- dropdown items -->
    </div>
</div>
```

The `@click.away` directive closes the dropdown when the user clicks anywhere outside it. The `x-transition` directives add a scale+fade animation.

### 6.5.3 Modal Dialogs

The `<x-modal>` component wraps Alpine.js show/hide logic:

```html
<div x-data="{ show: false }"
     x-show="show"
     x-on:open-modal.window="show = ($event.detail === 'confirm-deletion')"
     x-on:close.stop="show = false"
     class="fixed inset-0 z-50">

    <!-- Backdrop -->
    <div x-show="show" x-transition:enter="ease-out duration-300"
         class="fixed inset-0 bg-gray-500/75" @click="show = false">
    </div>

    <!-- Modal panel -->
    <div x-show="show" x-transition:enter="ease-out duration-300"
         x-transition:enter-start="opacity-0 translate-y-4 sm:scale-95"
         x-transition:enter-end="opacity-100 translate-y-0 sm:scale-100"
         class="bg-white rounded-lg shadow-xl max-w-lg mx-auto mt-20 p-6">
        {{ $slot }}
    </div>
</div>
```

Modals are opened by dispatching a browser event: `$dispatch('open-modal', 'confirm-deletion')`. The modal component listens for this event and shows itself if the name matches.

### 6.5.4 Tab Selection

```html
<div x-data="{ activeTab: 'info' }">
    <div class="border-b flex gap-4">
        <button @click="activeTab = 'info'"
```

### 6.5.5 AI Advisor Chat

The mahasiswa AI Advisor page uses a dedicated Alpine.js component function:

```html
<div x-data="aiAdvisor()">
    <!-- Message list -->
    <div x-ref="messages" class="flex-1 overflow-y-auto p-4 space-y-4">
        <template x-for="message in messages" :key="message.id">
            <div :class="message.role === 'user' ? 'justify-end' : 'justify-start'"
                 class="flex">
                <div :class="message.role === 'user'
                    ? 'bg-siakad-primary text-white'
                    : 'bg-gray-100 dark:bg-gray-700'"
                     class="rounded-lg px-4 py-2 max-w-[80%]"
                     x-html="message.content">
                </div>
            </div>
        </template>
        <div x-show="loading" class="flex justify-start">
            <div class="bg-gray-100 rounded-lg px-4 py-2">Mengetik...</div>
        </div>
    </div>

    <!-- Input -->
    <form @submit.prevent="sendMessage" class="p-4 border-t">
        <input x-model="input" type="text" placeholder="Tanya sesuatu..."
               class="w-full rounded-lg border px-4 py-2" />
    </form>
</div>
```

```js
function aiAdvisor() {
    return {
        messages: [],
        input: '',
        loading: false,

        async sendMessage() {
            if (!this.input.trim()) return;

            const userMessage = { id: Date.now(), role: 'user', content: this.input };
            this.messages.push(userMessage);
            this.input = '';
            this.loading = true;

            const response = await fetch('/mahasiswa/ai-advisor/chat', {
                method: 'POST',
                headers: {
                    'Content-Type': 'application/json',
                    'X-CSRF-TOKEN': document.querySelector('meta[name="csrf-token"]').content
                },
                body: JSON.stringify({ message: userMessage.content })
            });

            const data = await response.json();
            this.messages.push({ id: Date.now() + 1, role: 'assistant', content: data.response });
            this.loading = false;

            this.$nextTick(() => {
                this.$refs.messages.scrollTop = this.$refs.messages.scrollHeight;
            });
        }
    }
}
```

The chat auto-scrolls to the latest message after each response using `$nextTick` and `scrollTop`. Responses are rendered as HTML (`x-html`) to support Markdown formatting from the AI backend.

## 6.6 JavaScript Architecture

### 6.6.1 Entry Points

**`resources/js/app.js`**

```js
import Alpine from 'alpinejs';

window.Alpine = Alpine;
Alpine.start();
```

This file does nothing beyond importing Alpine.js and starting it. No custom Alpine plugins, stores, or global components are registered.

**`resources/js/bootstrap.js`**

```js
import axios from 'axios';

window.axios = axios;
window.axios.defaults.headers.common['X-Requested-With'] = 'XMLHttpRequest';

const token = document.head.querySelector('meta[name="csrf-token"]');
if (token) {
    window.axios.defaults.headers.common['X-CSRF-TOKEN'] = token.content;
}
```

This configures Axios globally: the CSRF token is read from a `<meta>` tag in the HTML `<head>` and attached to every outgoing request. The `X-Requested-With` header ensures Laravel recognizes these as AJAX requests.

### 6.6.2 Vanilla JavaScript Patterns

Beyond Alpine.js, several pages use raw JavaScript for interactions that don't require reactive state:

**Modal toggle via classList:**
```js
document.getElementById('createModal').classList.remove('hidden');
document.getElementById('createModal').classList.add('hidden');
```

**Form population on edit:**
```js
document.querySelectorAll('.btn-edit').forEach(btn => {
    btn.addEventListener('click', function() {
        document.getElementById('edit_nama').value = this.dataset.nama;
        document.getElementById('edit_kode').value = this.dataset.kode;
        document.getElementById('editForm').action = '/admin/prodi/' + this.dataset.id;
        document.getElementById('editModal').classList.remove('hidden');
    });
});
```

Data is stored in `data-*` attributes on the edit button, then read and injected into form fields when clicked.

**Cascading dropdowns:**
```js
document.getElementById('fakultas_id').addEventListener('change', function() {
    fetch('/api/prodi?fakultas_id=' + this.value)
        .then(res => res.json())
        .then(data => {
            const prodiSelect = document.getElementById('prodi_id');
            prodiSelect.innerHTML = '<option value="">Pilih Prodi</option>';
            data.forEach(prodi => {
                prodiSelect.innerHTML += `<option value="${prodi.id}">${prodi.nama}</option>`;
            });
        });
});
```

This pattern appears in admin forms where selecting a Fakultas filters the available Prodi options.

**Debounced search:**
```js
let searchTimeout;
document.getElementById('search').addEventListener('input', function() {
    clearTimeout(searchTimeout);
    searchTimeout = setTimeout(() => {
        // perform search or filter table rows
    }, 300);
});
```

A 300ms debounce prevents excessive requests or DOM updates while the user is still typing.

## 6.7 Chart.js Integration

Chart.js is loaded from CDN only on dashboard pages — it is not part of the Vite bundle. This keeps the main bundle size small for pages that don't need charts.

### Loading

```blade
@push('scripts')
<script src="https://cdn.jsdelivr.net/npm/chart.js"></script>
<script>
    // chart initialization code
</script>
@endpush
```

The `@push('scripts')` directive appends the script tag to the `@stack('scripts')` placeholder in the layout, so Chart.js is only loaded when the page actually needs it.

### Admin Dashboard Charts

Bar charts show student and class counts grouped by program (prodi):

```js
const ctx = document.getElementById('chartMahasiswaPerProdi').getContext('2d');
new Chart(ctx, {
```

## 6.9 Blade Template Patterns and Conventions

### 6.9.1 Layout Inheritance

Every page extends a layout using the `<x-app-layout>` component syntax:

```blade
<x-app-layout>
    <x-slot name="header">
        <h2 class="text-xl font-semibold">Dashboard</h2>
    </x-slot>

    {{-- page content --}}
</x-app-layout>
```

The `{{ $slot }}` in `app.blade.php` receives the page content. Named slots like `header` are inserted into designated areas of the layout.

### 6.9.2 Conditional Rendering

```blade
{{-- Authentication-based --}}
@auth
    <span>Welcome, {{ auth()->user()->name }}</span>
@endauth

@guest
    <a href="{{ route('login') }}">Login</a>
@endguest

{{-- Role-based (using direct role check, NOT policies) --}}
@if(auth()->user()->role === 'admin')
    <a href="/admin/users">Manage Users</a>
@endif
```

Authorization policies (`@can`/`@cannot`) are not used despite policy files existing — they are empty stubs.

### 6.9.3 Loops with Empty State

```blade
@forelse($mahasiswa as $mhs)
    <tr>
        <td>{{ $loop->iteration }}</td>
        <td>{{ $mhs->nama }}</td>
        <td>{{ $mhs->nim }}</td>
    </tr>
@empty
    <tr>
        <td colspan="3" class="text-center text-gray-500 py-4">
            Tidak ada data mahasiswa.
        </td>
    </tr>
@endforelse
```

### 6.9.4 Partial Includes

Views are broken into partials for reuse and readability:

```blade
{{-- Load a partial with data --}}
@include('dosen.jadwal._table', ['jadwals' => $jadwals])

{{-- Notification component in the header --}}
@include('components.notification-dropdown')
```

### 6.9.5 Per-Page Scripts and Styles

Pages that need additional JavaScript (e.g., Chart.js, custom handlers) use the `@push` / `@stack` system:

```blade
{{-- In the page view --}}
@push('scripts')
<script>
    // page-specific JavaScript
</script>
@endpush

{{-- In the layout (app.blade.php) --}}
<body>
    {{ $slot }}
    @stack('scripts')
</body>
```

This keeps per-page scripts out of the global bundle and loads them only when needed.

### 6.9.6 Component Props

Components accept props using the `@props` directive:

```blade
{{-- text-input.blade.php --}}
@props(['disabled' => false])

<input {{ $disabled ? 'disabled' : '' }} {!! $attributes->merge([
    'class' => 'border-gray-300 focus:border-siakad-primary focus:ring-siakad-primary rounded-md shadow-sm'
]) !!}>
```

The `$attributes->merge()` call allows parent templates to add classes or attributes that get merged with the defaults.

## 6.10 Build Pipeline

### Vite Configuration

```js
// vite.config.js
import { defineConfig } from 'vite';
import laravel from 'laravel-vite-plugin';

export default defineConfig({
    plugins: [
        laravel({
            input: ['resources/css/app.css', 'resources/js/app.js'],
            refresh: true,
        }),
    ],
});
```

Two entry points are defined:
1. `resources/css/app.css` — Tailwind directives, custom properties, utility classes
2. `resources/js/app.js` — Alpine.js initialization

The `refresh: true` setting triggers full page reloads during development when Blade files change.

### Build Process

```
┌─────────────────────────────────────────────────────┐
│                  BUILD PIPELINE                      │
│                                                      │
│  resources/css/app.css ─┐                            │
│                         ├──► Vite (via Bun) ──► public/build/  │
│  resources/js/app.js ───┘        │                   │
│                                  │                   │
│                           ┌──────┴──────┐            │
│                           │  Processes  │            │
│                           ├─────────────┤            │
│                           │ Tailwind    │            │
│                           │ PostCSS     │            │
│                           │ JS bundling │            │
│                           │ Minification│            │
│                           └─────────────┘            │
│                                                      │
│  Output: public/build/manifest.json                  │
│          public/build/assets/app-[hash].css           │
│          public/build/assets/app-[hash].js            │
└─────────────────────────────────────────────────────┘
```

| Command | Purpose |
|---|---|
| `bun run dev` | Start Vite dev server with HMR |
| `bun run build` | Production build with minification |

In production, `@vite(['resources/css/app.css', 'resources/js/app.js'])` reads `public/build/manifest.json` to resolve hashed filenames. In development, it points to the Vite dev server for hot module replacement.

### What Is NOT Bundled

- **Chart.js** — Loaded via CDN on pages that need it (`@push('scripts')`)
- **Per-page inline scripts** — Written directly in Blade views via `@push('scripts')`
- **Alpine.js component functions** (like `aiAdvisor()`) — Defined inline in Blade templates, not as separate JS modules

## 6.11 Mobile Responsiveness Approach

SIAKAD does not use a separate mobile site or a responsive SPA framework. Responsiveness is handled entirely through Tailwind CSS responsive prefixes and conditional rendering.

### Responsive Strategies

| Strategy | Implementation |
|---|---|
| Sidebar collapse | Hidden on mobile (`hidden lg:block`), replaced by bottom nav or hamburger overlay |
| Mobile bottom nav | Visible only on small screens (`block sm:hidden`), mahasiswa role only |
| Table horizontal scroll | Tables wrapped in `overflow-x-auto` containers |
| Grid column reduction | Desktop: `grid-cols-4`, Tablet: `md:grid-cols-2`, Mobile: `grid-cols-1` |
| Font size scaling | `text-sm` base on mobile, `md:text-base` or `lg:text-lg` on larger screens |
| Card stacking | Cards arranged in flexbox/grid that wraps vertically on narrow screens |

### Mobile Navigation by Role

```
┌─────────────────────────────────────────────────────┐
│ Mahasiswa (mobile):                                  │
│   Bottom navigation bar (5 icons, always visible)    │
│   Sidebar: hidden                                    │
│                                                      │
│ Dosen (mobile):                                      │
│   Hamburger button → sidebar overlay                 │
│   No bottom navigation bar                           │
│                                                      │
│ Admin (mobile):                                      │
│   Hamburger button → sidebar overlay                 │
│   No bottom navigation bar                           │
└─────────────────────────────────────────────────────┘
```

The mahasiswa role gets a dedicated mobile bottom navigation because mahasiswa are the most likely users on mobile devices. Dosen and admin roles are expected to use desktop browsers for most tasks.

### Responsive Table Pattern

Admin data tables use horizontal scrolling on small screens rather than hiding columns:

```blade
<div class="overflow-x-auto">
    <table class="min-w-full divide-y divide-gray-200">
        <thead>
            <tr>
                <th class="px-6 py-3 text-left text-xs font-medium text-gray-500 uppercase">No</th>
                <th class="px-6 py-3 text-left text-xs font-medium text-gray-500 uppercase">Nama</th>
                {{-- more columns --}}
            </tr>
        </thead>
        <tbody class="divide-y divide-gray-200">
            {{-- rows --}}
        </tbody>
    </table>
</div>
```

## 6.12 Dark Mode Implementation

Dark mode is implemented using Tailwind's `class` strategy — the `.dark` class on `<html>` activates dark mode variants throughout the application.

### Toggle Mechanism

```js
// In the header bar
function toggleDarkMode() {
    const html = document.documentElement;
    html.classList.toggle('dark');
    localStorage.setItem('darkMode', html.classList.contains('dark'));
}

// On page load — restore preference
if (localStorage.getItem('darkMode') === 'true') {
    document.documentElement.classList.add('dark');
}
```

The toggle is a button in the top header bar. The user's preference is stored in `localStorage` under the key `darkMode` and restored on every page load before the page renders (to prevent flash of wrong theme).

### Dark Mode in Tailwind Classes

Every element that needs a dark variant uses the `dark:` prefix:

```html
<div class="bg-white dark:bg-gray-800 text-gray-900 dark:text-gray-100">
    <h2 class="text-gray-700 dark:text-gray-300">Title</h2>
    <p class="text-gray-600 dark:text-gray-400">Description</p>
</div>
```

### Dark Mode Coverage

| UI Element | Light Mode | Dark Mode |
|---|---|---|
| Page background | `bg-gray-100` | `dark:bg-gray-900` |
| Sidebar | `bg-white` | `dark:bg-gray-800` |
| Cards | `bg-white` | `dark:bg-gray-800` |
| Text (primary) | `text-gray-900` | `dark:text-gray-100` |
| Text (secondary) | `text-gray-600` | `dark:text-gray-400` |
| Table rows (even) | `even:bg-gray-50` | `dark:even:bg-gray-700` |
| Borders | `border-gray-200` | `dark:border-gray-700` |
| Input fields | `bg-white border-gray-300` | `dark:bg-gray-700 dark:border-gray-600` |
| Chart colors | Blue `#1e40af` | Light blue `#60a5fa` |

### CSS Custom Properties in Dark Mode

The CSS variables defined in `:root` are overridden when `.dark` is applied:

```css
:root {
    --siakad-primary: #1e40af;
}

.dark {
    --siakad-primary: #3b82f6;  /* lighter shade for contrast on dark backgrounds */
}
```

This means components using `var(--siakad-primary)` automatically adjust without needing Tailwind `dark:` classes.

### Chart.js Dark Mode

Charts detect the current theme and adjust colors at render time:

```js
const isDarkMode = document.documentElement.classList.contains('dark');

const chartConfig = {
    // ...
    options: {
        plugins: {
            legend: {
                labels: { color: isDarkMode ? '#e5e7eb' : '#1f2937' }
            }
        },
        scales: {
            x: { ticks: { color: isDarkMode ? '#9ca3af' : '#4b5563' },
                 grid: { color: isDarkMode ? '#374151' : '#e5e7eb' } },
            y: { ticks: { color: isDarkMode ? '#9ca3af' : '#4b5563' },
                 grid: { color: isDarkMode ? '#374151' : '#e5e7eb' } }
        }
    }
};
```

Charts are not reactive to dark mode toggle — they render with the theme that was active at page load. Toggling dark mode after the page loads does not re-render existing charts.

| Aspect | Approach |
|---|---|
| Rendering | Server-side (Blade), no SPA |
| Reactivity | Alpine.js v3 (inline `x-data` components) |
| Styling | Tailwind CSS v3 + custom CSS properties |
| Bundling | Vite 7 via Bun, two entry points |
| Charts | Chart.js via CDN, dashboard pages only |
| Dark mode | `class` strategy with `localStorage` persistence |
