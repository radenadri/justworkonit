# Chapter 3: Database Schema and Eloquent Models

## 3.1 Schema Overview

SIAKAD uses a MySQL/MariaDB database with 38 migration files producing approximately 30 application tables plus 5 Spatie permission tables. The schema divides into eight functional domains:

| Domain | Tables | Purpose |
|--------|--------|---------|
| Core | `users`, `fakultas`, `prodi`, `dosen`, `mahasiswa`, `admin_fakultas` | Identity, organizational hierarchy, user profiles |
| Academic | `tahun_akademik`, `mata_kuliah`, `kelas`, `jadwal_kuliah`, `ruangan` | Courses, scheduling, room allocation |
| KRS | `krs`, `krs_detail` | Course registration per semester |
| Grading | `nilai` | Final grades per student per class |
| Attendance | `pertemuan`, `presensi`, `kehadiran_dosen` | Session tracking, student/lecturer attendance |
| LMS | `materi`, `tugas`, `tugas_submission` | Learning materials, assignments, submissions |
| Thesis & Internship | `skripsi`, `bimbingan`, `kerja_praktek`, `logbook_kp` | Thesis management, internship tracking |
| Supporting | `notifications`, `activity_logs`, `ai_conversations`, permission tables | Audit trail, notifications, AI chatbot, RBAC |

## 3.3 Table-by-Table Documentation

### 3.3.1 Core Domain

#### `users`

Central authentication table. Every person in the system—superadmin, faculty admin, lecturer, or student—has exactly one row here.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `bigint unsigned` | PK, auto-increment | |
| `name` | `string` | NOT NULL | Display name |
| `email` | `string` | NOT NULL, UNIQUE | Login identifier |
| `email_verified_at` | `timestamp` | NULLABLE | Laravel email verification |
| `password` | `string` | NOT NULL | Bcrypt hash |
| `role` | `enum` | NOT NULL | `superadmin`, `admin_fakultas`, `dosen`, `mahasiswa` |
| `remember_token` | `string(100)` | NULLABLE | "Remember me" session token |
| `created_at` | `timestamp` | NULLABLE | |
| `updated_at` | `timestamp` | NULLABLE | |

**Indexes:** unique on `email`

**Design notes:** The `role` column is an enum stored directly on the user row, providing fast role checks without joins. The system also uses Spatie's `model_has_roles` table, creating a dual-role system. The `HasRole` trait on the User model exposes convenience methods (`isSuperadmin()`, `isDosen()`, etc.) that check the enum column, not Spatie roles.

---

#### `fakultas`

Top-level organizational unit. A university faculty (e.g., "Fakultas Teknik").

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `bigint unsigned` | PK, auto-increment | |
| `nama` | `string` | NOT NULL | Full faculty name |
| `kode_fakultas` | `string` | NOT NULL, UNIQUE | Short code (e.g., "FT") |
| `deskripsi` | `text` | NULLABLE | Optional description |
| `created_at` | `timestamp` | NULLABLE | |
| `updated_at` | `timestamp` | NULLABLE | |

**Indexes:** unique on `kode_fakultas`

**Referenced by:** `prodi.fakultas_id`, `dosen.fakultas_id`, `mahasiswa.fakultas_id`, `admin_fakultas.fakultas_id`, `users.fakultas_id` (though `users` does not have an FK column — the accessor `getFakultasIdAttribute()` resolves it through related models).

Study program (Program Studi). Sits under a faculty. All courses and students belong to a prodi.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `bigint unsigned` | PK, auto-increment | |
| `nama` | `string` | NOT NULL | Program name |
| `kode_prodi` | `string` | NOT NULL, UNIQUE | Short code (e.g., "SI") |
| `fakultas_id` | `bigint unsigned` | FK → `fakultas(id)` CASCADE | Parent faculty |
| `jenjang` | `enum` | NOT NULL | `S1`, `S2`, `S3`, `D3`, `D4` |
| `akreditasi` | `string` | NULLABLE | Accreditation grade (e.g., "A", "Unggul") |
| `deskripsi` | `text` | NULLABLE | |
| `created_at` | `timestamp` | NULLABLE | |
| `updated_at` | `timestamp` | NULLABLE | |

**Indexes:** unique on `kode_prodi`
**Foreign keys:** `fakultas_id` → `fakultas(id)` ON DELETE CASCADE

#### `dosen`

Lecturer profile. One-to-one with `users`. A dosen belongs to a faculty and optionally to a prodi.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `bigint unsigned` | PK, auto-increment | |
| `user_id` | `bigint unsigned` | FK → `users(id)` CASCADE, UNIQUE | Auth account link |
| `nidn` | `string` | NOT NULL, UNIQUE | National lecturer ID number |
| `nama` | `string` | NOT NULL | Full name |
| `email` | `string` | NULLABLE | Separate from `users.email`; may be institutional email |
| `no_telepon` | `string` | NULLABLE | |
| `alamat` | `text` | NULLABLE | |
| `fakultas_id` | `bigint unsigned` | FK → `fakultas(id)` CASCADE | |
| `prodi_id` | `bigint unsigned` | FK → `prodi(id)` SET NULL, NULLABLE | |
| `jabatan_fungsional` | `string` | NULLABLE | Academic rank (Lektor, Guru Besar, etc.) |
| `status` | `enum` | NOT NULL, default `aktif` | `aktif`, `cuti`, `nonaktif` |
| `created_at` | `timestamp` | NULLABLE | |
| `updated_at` | `timestamp` | NULLABLE | |

**Indexes:** unique on `nidn`, unique on `user_id`
**Foreign keys:**
- `user_id` → `users(id)` ON DELETE CASCADE
- `fakultas_id` → `fakultas(id)` ON DELETE CASCADE
- `prodi_id` → `prodi(id)` ON DELETE SET NULL

**Design notes:** `prodi_id` is nullable with SET NULL because some lecturers serve at the faculty level without assignment to a specific program. The `email` field here is separate from `users.email` — this allows an institutional email distinct from the login email.

#### `mahasiswa`

Student profile. One-to-one with `users`. Contains academic and personal data.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `bigint unsigned` | PK, auto-increment | |
| `user_id` | `bigint unsigned` | FK → `users(id)` CASCADE, UNIQUE | Auth account link |
| `nim` | `string` | NOT NULL, UNIQUE | Student ID number |
| `nama` | `string` | NOT NULL | Full name |
| `email` | `string` | NULLABLE | Institutional email |
| `no_telepon` | `string` | NULLABLE | |
| `alamat` | `text` | NULLABLE | |
| `tanggal_lahir` | `date` | NULLABLE | Date of birth |
| `jenis_kelamin` | `enum` | NULLABLE | `L` (male), `P` (female) |
| `fakultas_id` | `bigint unsigned` | FK → `fakultas(id)` CASCADE | |
| `prodi_id` | `bigint unsigned` | FK → `prodi(id)` CASCADE | |
| `semester` | `integer` | NOT NULL, default `1` | Current semester number |
| `status` | `enum` | NOT NULL, default `aktif` | `aktif`, `cuti`, `lulus`, `do`, `mengundurkan_diri` |
| `dosen_pa_id` | `bigint unsigned` | FK → `dosen(id)` SET NULL, NULLABLE | Academic advisor |
| `angkatan` | `string` | NULLABLE | Enrollment year (e.g., "2023") |
| `created_at` | `timestamp` | NULLABLE | |
| `updated_at` | `timestamp` | NULLABLE | |

**Indexes:** unique on `nim`, unique on `user_id`
**Foreign keys:**
- `user_id` → `users(id)` ON DELETE CASCADE
- `fakultas_id` → `fakultas(id)` ON DELETE CASCADE
- `prodi_id` → `prodi(id)` ON DELETE CASCADE
- `dosen_pa_id` → `dosen(id)` ON DELETE SET NULL

**Design notes:** Unlike `dosen.prodi_id`, the student's `prodi_id` is NOT nullable — every student must belong to a program. The `dosen_pa_id` column links to the student's academic advisor (Pembimbing Akademik). SET NULL on delete prevents cascading student deletion when an advisor record is removed.

| `user_id` | `bigint unsigned` | FK → `users(id)` CASCADE | |
| `fakultas_id` | `bigint unsigned` | FK → `fakultas(id)` CASCADE | |
| `created_at` | `timestamp` | NULLABLE | |
| `updated_at` | `timestamp` | NULLABLE | |

**Foreign keys:**
- `user_id` → `users(id)` ON DELETE CASCADE

### 3.3.2 Academic Domain

#### `tahun_akademik`

Academic year/semester periods. Controls which semester is active for KRS, grading, and scheduling.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `bigint unsigned` | PK, auto-increment | |
| `tahun` | `string` | NOT NULL | Format: "2023/2024" |
| `semester` | `enum` | NOT NULL | `ganjil` (odd), `genap` (even) |
| `is_active` | `boolean` | NOT NULL, default `false` | Only one should be `true` at a time |
| `tanggal_mulai` | `date` | NOT NULL | Semester start date |
| `tanggal_selesai` | `date` | NOT NULL | Semester end date |
| `created_at` | `timestamp` | NULLABLE | |
| `updated_at` | `timestamp` | NULLABLE | |

**Scopes:** The Eloquent model defines an `active()` scope to query only the currently active academic period.

**Business rule:** Although the database does not enforce a single active row via unique constraint, application logic should set only one row to `is_active = true` at a time.

| `prodi_id` | `bigint unsigned` | FK → `prodi(id)` CASCADE | Owning program |

| `updated_at` | `timestamp` | NULLABLE | |

**Indexes:** unique on `kode_mk`
**Foreign keys:** `prodi_id` → `prodi(id)` ON DELETE CASCADE

#### `kelas`

A class section: one offering of a course in a given semester, taught by one lecturer.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `bigint unsigned` | PK, auto-increment | |
| `kode_kelas` | `string` | NOT NULL | Section code (e.g., "A", "B") |
| `mata_kuliah_id` | `bigint unsigned` | FK → `mata_kuliah(id)` CASCADE | |
| `dosen_id` | `bigint unsigned` | FK → `dosen(id)` CASCADE | Teaching lecturer |
| `tahun_akademik_id` | `bigint unsigned` | FK → `tahun_akademik(id)` CASCADE | Academic period |
| `kapasitas` | `integer` | NOT NULL, default `40` | Max enrollment |
| `jumlah_mahasiswa` | `integer` | NOT NULL, default `0` | Current enrollment count |
| `hari` | `enum` | NOT NULL | `senin`, `selasa`, `rabu`, `kamis`, `jumat` |
| `jam_mulai` | `time` | NOT NULL | Start time |
| `jam_selesai` | `time` | NOT NULL | End time |
| `ruangan` | `string` | NULLABLE | Room name (plain text, not FK) |
| `created_at` | `timestamp` | NULLABLE | |
| `updated_at` | `timestamp` | NULLABLE | |

**Foreign keys:**
- `mata_kuliah_id` → `mata_kuliah(id)` ON DELETE CASCADE
- `dosen_id` → `dosen(id)` ON DELETE CASCADE
- `tahun_akademik_id` → `tahun_akademik(id)` ON DELETE CASCADE

**Design notes:** The `kelas` table stores both scheduling data (`hari`, `jam_mulai`, `jam_selesai`) and capacity tracking (`kapasitas`, `jumlah_mahasiswa`). The `ruangan` column here is a plain string, separate from the `ruangan` table. The `jadwal_kuliah` table provides a more normalized scheduling model with an FK to the `ruangan` table. This duplication suggests an incremental migration: `kelas` held the original schedule, and `jadwal_kuliah` was added later for richer scheduling with room management.

### 3.3.3 KRS Domain (Course Registration)

#### `krs`

KRS (Kartu Rencana Studi) header. One per student per academic period. Tracks approval workflow.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `bigint unsigned` | PK, auto-increment | |
| `mahasiswa_id` | `bigint unsigned` | FK → `mahasiswa(id)` CASCADE | |
| `tahun_akademik_id` | `bigint unsigned` | FK → `tahun_akademik(id)` CASCADE | |
| `status` | `enum` | NOT NULL, default `draft` | `draft`, `pending`, `approved`, `rejected` |
| `catatan` | `text` | NULLABLE | Reviewer notes |
| `approved_by` | `bigint unsigned` | FK → `users(id)` SET NULL, NULLABLE | Approver (advisor/admin) |
| `approved_at` | `timestamp` | NULLABLE | Approval timestamp |
| `total_sks` | `integer` | NOT NULL, default `0` | Sum of selected course credits |
| `created_at` | `timestamp` | NULLABLE | |
| `updated_at` | `timestamp` | NULLABLE | |

**Indexes:** unique composite on `(mahasiswa_id, tahun_akademik_id)`
**Foreign keys:**
- `mahasiswa_id` → `mahasiswa(id)` ON DELETE CASCADE
- `tahun_akademik_id` → `tahun_akademik(id)` ON DELETE CASCADE
- `approved_by` → `users(id)` ON DELETE SET NULL

**Composite unique constraint purpose:** A student can only have one KRS per academic period. Prevents duplicate registrations for the same semester.

## 3.5 Composite Unique Constraints

The database uses composite unique constraints to enforce business rules at the database level:

| Table | Columns | Business Rule |
|-------|---------|---------------|
| `krs` | `(mahasiswa_id, tahun_akademik_id)` | One KRS per student per academic period |
| `krs_detail` | `(krs_id, kelas_id)` | Cannot register same class twice in one KRS |
| `nilai` | `(mahasiswa_id, kelas_id)` | One grade record per student per class |
| `pertemuan` | `(jadwal_kuliah_id, pertemuan_ke)` | Unique session numbers within a schedule |
| `presensi` | `(pertemuan_id, mahasiswa_id)` | One attendance record per student per session |
| `kehadiran_dosen` | `(dosen_id, jadwal_kuliah_id, tanggal)` | One lecturer attendance per schedule per date |
| `tugas_submission` | `(tugas_id, mahasiswa_id)` | One submission per student per assignment |

These constraints prevent duplicate data at the database level regardless of application-layer validation.

## 3.8 Business Rules Encoded in the Database

### 3.8.1 Grade Conversion Table

The application converts numeric scores to letter grades and grade points using this scale:

| Numeric Range | Letter Grade | Grade Point (Bobot) |
|--------------|-------------|-------------------|
| 85 – 100 | A | 4.00 |
| 80 – 84 | A- | 3.75 |
| 75 – 79 | B+ | 3.50 |
| 70 – 74 | B | 3.00 |
| 65 – 69 | C+ | 2.50 |
| 60 – 64 | C | 2.00 |
| 55 – 59 | D | 1.00 |
| 0 – 54 | E | 0.00 |

This conversion is applied in application logic (not stored procedures). The `nilai` table stores all three representations: `nilai_angka`, `nilai_huruf`, and `bobot`.

### 3.8.2 SKS Limits per Semester

The maximum SKS (credit hours) a student can register for depends on their IPS (Indeks Prestasi Semester — semester GPA):

| IPS Range | Max SKS Allowed |
|-----------|----------------|
| ≥ 3.51 | 24 |
| 3.01 – 3.50 | 22 |
| 2.51 – 3.00 | 20 |
| 2.01 – 2.50 | 18 |
| < 2.00 | 14 |

This rule is enforced during KRS creation. The `krs.total_sks` column tracks the running total. First-semester students (no prior IPS) typically receive a default allowance.

### 3.8.3 Graduation Requirements

For the Sistem Informasi program at UNRI (Universitas Riau):

- **Total SKS required:** 144
- Student status changes to `lulus` upon completion

### 3.8.4 Status Lifecycles

#### KRS Lifecycle

```
draft ──→ pending ──→ approved
                  └──→ rejected
```

- **draft**: Student is selecting courses, can add/remove freely
- **pending**: Student submitted for advisor review
- **approved**: Advisor/admin approved the registration
- **rejected**: Advisor/admin rejected; student revises and resubmits

#### Grade (Nilai) Lifecycle

```
draft ──→ submitted ──→ final
```

- **draft**: Lecturer is entering grades, can modify freely
- **submitted**: Grades submitted for review
- **final**: Grades locked, visible to students, counted toward GPA

#### Thesis (Skripsi) Lifecycle

```
pengajuan ──→ bimbingan ──→ seminar_proposal ──→ penelitian
                                                      │
              selesai ←── revisi ←── sidang ←── seminar_hasil
```

| Status | English | Meaning |
|--------|---------|---------|
| `pengajuan` | Proposal submission | Initial topic/advisor request |
| `bimbingan` | Advising | Active consultation with advisor |
| `seminar_proposal` | Proposal seminar | Public proposal defense |
| `penelitian` | Research | Active research/implementation |
| `seminar_hasil` | Results seminar | Presentation of findings |
| `sidang` | Final defense | Thesis defense examination |
| `revisi` | Revision | Post-defense corrections |
| `selesai` | Completed | Thesis accepted and archived |

#### Internship (Kerja Praktek) Lifecycle

```
pengajuan ──→ berlangsung ──→ selesai
         └──→ batal
```

| Status | English | Meaning |
|--------|---------|---------|
| `pengajuan` | Application | Student applied for internship |
| `berlangsung` | In progress | Currently at the company |
| `selesai` | Completed | Internship finished, report submitted |
| `batal` | Cancelled | Internship cancelled |

#### Student Status Transitions

```
aktif ──→ cuti ──→ aktif (return from leave)
     └──→ lulus (graduated)
     └──→ do (dropped out)
     └──→ mengundurkan_diri (resigned)
```

### 3.8.5 Attendance Rules

- **Minimum attendance:** 75% of total sessions
- Students below 75% attendance may be barred from final exams
- Four attendance statuses with different implications:

| Status | Indonesian | Counts as Present | Notes |
|--------|-----------|-------------------|-------|
| `hadir` | Present | Yes | Normal attendance |
| `sakit` | Sick | Conditional | With medical certificate |
| `izin` | Excused | Conditional | Pre-approved absence |
| `alpa` | Absent | No | Unexcused absence |

### 3.8.6 Class Capacity Enforcement

The `kelas` table tracks `kapasitas` (max capacity) and `jumlah_mahasiswa` (current count). Application logic checks `jumlah_mahasiswa < kapasitas` before allowing KRS enrollment. The counter is maintained via application code (not database triggers).

## 3.9 Schema Design Observations

### Denormalization Choices

1. **Schedule duplication:** Both `kelas` and `jadwal_kuliah` store day/time data. The `kelas` table holds inline schedule fields (`hari`, `jam_mulai`, `jam_selesai`, `ruangan` as string), while `jadwal_kuliah` provides a normalized alternative with FK to `ruangan`. This suggests an evolutionary design where `jadwal_kuliah` replaced the inline approach but the old columns were retained for backward compatibility.

2. **Name duplication:** The `nama` column appears on both `users` and `dosen`/`mahasiswa` tables. The profile tables (`dosen`, `mahasiswa`) carry their own `nama` for display purposes, potentially differing from the `users.name` used for authentication.

3. **Counter caching:** `kelas.jumlah_mahasiswa` is a denormalized counter. The actual count is derivable from `krs_detail` records pointing to this class, but the counter avoids a COUNT query on every enrollment check.

### Cascade Delete Strategy

| FK Delete Behavior | Used For |
|-------------------|----------|
| CASCADE | Core ownership (deleting parent removes children). Used for: mahasiswa→krs, kelas→jadwal_kuliah, skripsi→bimbingan, etc. |
| SET NULL | Optional relationships. Used for: dosen.prodi_id, mahasiswa.dosen_pa_id, skripsi.pembimbing2_id, krs.approved_by |

No RESTRICT constraints are used. This means deleting a faculty will cascade-delete all its programs, students, lecturers, and their associated academic records.

### Polymorphic Relationships

Two tables use polymorphic columns:
- `notifications` — `notifiable_type` + `notifiable_id` (Laravel standard)
- `activity_logs` — `model_type` + `model_id` (custom audit log)

### Missing Database-Level Constraints

The following rules are enforced only in application code:
- Only one `tahun_akademik` can be `is_active = true` (no partial unique index)
- SKS limit per KRS based on IPS
- 75% attendance minimum
- Grade conversion formula
- Class capacity limits
