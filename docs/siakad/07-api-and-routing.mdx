# Chapter 7: Routing Architecture

## 7.1 Overview

SIAKAD is a fully server-rendered Laravel application. Every user-facing endpoint returns an HTML response through Blade templates. There is no `routes/api.php`, no SPA frontend, and no token-based API authentication. The routing layer is organized by user role, with each role receiving its own directory of route files and its own URL prefix.

This chapter covers the file organization, middleware stack, route groups, naming conventions, rate limiting, and RESTful patterns used throughout the application.

**Total: 42 route files** across 4 directories (root, admin, dosen, mahasiswa).

The rationale for splitting routes into per-role directories:

1. **Middleware isolation** — Each directory inherits its parent group's middleware. A route file inside `routes/dosen/` never needs to declare `role:dosen` middleware; it is applied at the group level in `web.php`.
2. **Developer navigation** — When a developer works on the grading feature for lecturers, they open `routes/dosen/penilaian.php` directly. No scrolling through a monolithic 500-line route file.
3. **Merge conflict reduction** — Feature branches that modify admin routes do not conflict with branches modifying student routes.
4. **Access audit** — To audit every endpoint accessible to students, read the files in `routes/mahasiswa/`. No grep required.

### 7.3.2 Middleware Applied Per Group

| Route Group | Middleware Stack | Purpose |
|-------------|----------------|---------|
| Auth (guest) | `guest` | Prevent authenticated users from accessing login/register |
| Auth (protected) | `auth` | Require authentication for logout, password update |
| Admin | `auth`, `verified`, `role:admin_fakultas,superadmin` | Restrict to admin roles, require email verification |
| Dosen | `auth`, `verified`, `role:dosen` | Restrict to lecturers |
| Mahasiswa | `auth`, `verified`, `role:mahasiswa` | Restrict to students |
| Profile | `auth` | Any authenticated user can manage their profile |
| Notification | `auth` | Any authenticated user can manage notifications |
| Health | *(none)* | Public endpoint, no authentication required |

### 7.3.3 Custom Middleware

| Middleware | Class | Behavior |
|------------|-------|----------|
| `role` | `RoleMiddleware` | Checks `auth()->user()->role` against a comma-separated list of allowed roles. Returns 403 on mismatch. |
| `FakultasScope` | `FakultasScopeMiddleware` | For `admin_fakultas` users, automatically scopes Eloquent queries to the user's assigned `fakultas_id`. Superadmins bypass this scope. |

The `role` middleware accepts multiple roles as comma-separated values. In the admin group definition, `role:admin_fakultas,superadmin` grants access to both faculty admins and superadmins. A dosen attempting to access `/admin/dashboard` receives a 403 Forbidden response.

## 7.4 Root Route: Role-Based Redirect

The root URL (`/`) does not render a view. It inspects the authenticated user's role and issues a redirect:

```php
Route::get('/', function () {
    $user = auth()->user();

    return match ($user->role) {
        'superadmin', 'admin_fakultas' => redirect()->route('admin.dashboard'),
        'dosen'                        => redirect()->route('dosen.dashboard'),
        'mahasiswa'                    => redirect()->route('mahasiswa.dashboard'),
        default                        => redirect()->route('login'),
    };
});
```

Unauthenticated visitors are redirected to `/login`. There is no public-facing landing page.

## 7.6 Named Route Conventions

Every route in the application follows a consistent naming pattern:

```
{role}.{module}.{action}
```

Examples:

| Route Name | Pattern | Meaning |
|------------|---------|---------|
| `admin.mahasiswa.index` | role.module.action | Admin viewing student list |
| `dosen.presensi.store-pertemuan` | role.module.action | Lecturer creating a meeting session |
| `mahasiswa.krs.submit` | role.module.action | Student submitting KRS for approval |
| `admin.tahun-akademik.toggle` | role.module.action | Admin toggling academic year status |

Shared routes omit the role prefix: `profile.edit`, `notifications.read`, `health.check`.

Standard resource action names follow Laravel conventions: `index`, `create`, `store`, `show`, `edit`, `update`, `destroy`. Custom actions use hyphenated names: `store-pertemuan`, `approve-krs`, `bulk-update`.

**Usage in Blade templates:**

```php
{{-- Link to student dashboard --}}
<a href="{{ route('mahasiswa.dashboard') }}">Dashboard</a>

{{-- Form action for KRS submission --}}
<form action="{{ route('mahasiswa.krs.submit') }}" method="POST">

{{-- Admin edit link with route model binding --}}
<a href="{{ route('admin.mahasiswa.edit', $mahasiswa) }}">Edit</a>
```

This naming convention provides two guarantees:
1. Route names are globally unique (no collision between `admin.jadwal.index` and `dosen.jadwal.index`).
2. A developer can determine the required role and module from the route name alone.

## 7.7 Rate Limiting

Rate limiters are defined in `bootstrap/app.php` using Laravel's `RateLimiter` facade. Each limiter is keyed by the authenticated user's ID, so rate limits apply per-user rather than per-IP.

### 7.7.1 Rate Limiter Definitions

| Limiter Name | Limit | Window | Applied To |
|-------------|-------|--------|------------|
| `krs` | 10 requests | 1 minute | KRS store, destroy, submit |
| `penilaian` | 20 requests | 1 minute | Grade update, bulk grade update |
| `sensitive` | 30 requests | 1 minute | Admin write operations (store, update, destroy) |
| `ai-chat` | 10 requests | 1 minute | AI advisor chat endpoint |

### 7.7.2 Rationale for Each Limiter

**`krs` (10/min):** Course registration modifies enrollment records, adjusts class capacity counts, and triggers advisor notifications. Limiting to 10 requests per minute prevents accidental duplicate enrollments from rapid form resubmission and bounds the write load during peak registration periods (the first days of each semester).

**`penilaian` (20/min):** Grade updates are higher-frequency because lecturers may update individual student grades in succession. The 20/min limit accommodates normal grading workflows while preventing bulk scripted submissions.

**`sensitive` (30/min):** Admin write operations cover a wide range of CRUD actions. The limit is higher because admins perform batch data entry (e.g., creating 20 class sections for a new semester).

**`ai-chat` (10/min):** The AI advisor calls an external LLM API. Each request incurs cost and latency. The 10/min limit prevents runaway usage and keeps API costs predictable.

### 7.7.3 Rate Limiter Application

Rate limiters are applied as middleware on specific route definitions, not at the group level:

```php
// In routes/mahasiswa/krs.php
Route::middleware('throttle:krs')->group(function () {
    Route::post('/krs', [KrsController::class, 'store'])->name('mahasiswa.krs.store');
    Route::delete('/krs/{krsDetail}', [KrsController::class, 'destroy'])->name('mahasiswa.krs.destroy');
    Route::post('/krs/submit', [KrsController::class, 'submit'])->name('mahasiswa.krs.submit');
});

// The index (GET) route sits outside the throttle group:
Route::get('/krs', [KrsController::class, 'index'])->name('mahasiswa.krs.index');
```

Read-only `GET` routes are not rate-limited. Only state-changing operations (POST, PUT, DELETE) receive rate limits.

### 7.8.2 Resource + Extra Actions

Some admin resources extend the standard 7 actions with additional endpoints:

| Controller | Extra Action | Method | URI Suffix |
|------------|-------------|--------|------------|
| `Admin\MahasiswaController` | `export` | GET | `/export` |
| `Admin\DosenController` | `export` | GET | `/export` |
| `Admin\TahunAkademikController` | `toggleActive` | POST | `/{id}/toggle` |

The export routes must be registered *before* the resource routes to prevent `export` from being interpreted as a `{mahasiswa}` parameter by the `show` route.

### 7.8.3 Custom Route Patterns

Dosen and Mahasiswa controllers do not use `Route::resource()`. Their routes are defined individually because:

1. **Partial CRUD** — Students can store and destroy KRS items but not update them. Lecturers can view and update grades but not create or delete grade records.
2. **Non-standard nesting** — Attendance routes nest `pertemuan` (meetings) under `kelas` (classes), which does not map to Laravel's resource nesting.
3. **Action-specific endpoints** — KRS submit, KRS approve/reject, check-in/check-out are domain actions that do not correspond to REST verbs.

## 7.9 URL Patterns and RESTful Conventions

### 7.9.1 URL Structure

```
/{role}/{module}/{resource?}/{action?}
```

Examples:

```
/admin/mahasiswa                     → list all students
/admin/mahasiswa/create              → show create form
/admin/mahasiswa/42                  → show student #42
/admin/mahasiswa/42/edit             → show edit form for student #42

/dosen/presensi/5/pertemuan/3/input  → attendance input for class 5, meeting 3

/mahasiswa/krs                       → view current KRS
/mahasiswa/export/khs                → download KHS as PDF
```

### 7.9.2 Kebab-Case Resource Names

Multi-word resource names use kebab-case in URLs:

| URL Segment | Model |
|-------------|-------|
| `mata-kuliah` | `MataKuliah` |
| `tahun-akademik` | `TahunAkademik` |
| `kerja-praktek` | `KerjaPraktek` |
| `ai-advisor` | `AiAdvisor` |

This follows Laravel's default behavior when a hyphenated resource name is passed to `Route::resource()`.

### 7.9.3 Deviations from Strict REST

| Pattern | Standard REST | SIAKAD Pattern | Reason |
|---------|-------------|----------------|--------|
| KRS submit | `PATCH /krs/{id}` with status field | `POST /krs/submit` | Submit is a one-time state transition, not a partial update |
| KRS approve | `PATCH /krs/{id}` | `POST /bimbingan/krs/{krs}/approve` | Approval is a separate domain concept from KRS editing |
| Toggle active | `PATCH /tahun-akademik/{id}` | `POST /tahun-akademik/{id}/toggle` | Toggle implies idempotent state flip, distinct from a general update |
| Bulk grade | `PUT` per student | `POST /penilaian/{kelas}/bulk` | Batch operation avoids N individual requests |
| Check-in/out | `POST /kehadiran` with type field | `POST /kehadiran/checkin` and `/checkout` | Separate endpoints make the action explicit |

These deviations prioritize clarity of intent over strict RESTful purity. Each custom endpoint names the action being performed.

## 7.10 Route Model Binding

SIAKAD uses Laravel's implicit route model binding. Route parameters are type-hinted in controller methods, and Laravel automatically resolves the Eloquent model by its primary key.

### 7.10.1 Standard Binding

```php
// Route definition
Route::get('/admin/mahasiswa/{mahasiswa}', [MahasiswaController::class, 'show']);

// Controller method
public function show(Mahasiswa $mahasiswa): View
{
    return view('admin.mahasiswa.show', compact('mahasiswa'));
}
```

Laravel resolves `{mahasiswa}` to a `Mahasiswa` model instance by querying `WHERE id = ?`. If no matching record exists, Laravel returns a 404 response automatically.

### 7.10.2 Nested Binding

The attendance routes bind two models in a single URL:

```php
// Route: /dosen/presensi/{kelas}/pertemuan/{pertemuan}/input
public function inputPresensi(Kelas $kelas, Pertemuan $pertemuan): View
{
    // Both $kelas and $pertemuan are resolved automatically
}
```

### 7.10.3 Implicit Scoping

For nested resources, Laravel can enforce that the child model belongs to the parent. If `Pertemuan` has a `kelas_id` foreign key, Laravel validates that the resolved `Pertemuan` actually belongs to the specified `Kelas`. A mismatch results in a 404.

### 7.10.4 Auth-Scoped Binding

Some routes do not use route parameters at all but resolve the resource from the authenticated user:

```php
// Mahasiswa biodata — no {id} in URL
Route::get('/biodata', [BiodataController::class, 'index']);

// Controller resolves from auth
public function index(): View
{
    $mahasiswa = auth()->user()->mahasiswa;
    return view('mahasiswa.biodata.index', compact('mahasiswa'));
}
```

This pattern appears in biodata, KRS, KHS, and transcript routes. Students always view their own data; the URL does not expose an ID parameter.

## 7.11 Cross-Role Route Access Patterns

### 7.11.1 Access Matrix

```
                    ┌──────────┬──────────┬──────────┬────────────┐
                    │Superadmin│Admin Fak.│  Dosen   │ Mahasiswa  │
┌───────────────────┼──────────┼──────────┼──────────┼────────────┤
│ /admin/*          │    ✓     │    ✓*    │    ✗     │     ✗      │
│ /dosen/*          │    ✗     │    ✗     │    ✓     │     ✗      │
│ /mahasiswa/*      │    ✗     │    ✗     │    ✗     │     ✓      │
│ /profile          │    ✓     │    ✓     │    ✓     │     ✓      │
│ /notifications/*  │    ✓     │    ✓     │    ✓     │     ✓      │
│ /health           │    ✓     │    ✓     │    ✓     │     ✓      │
│ /login, /register │    ✗†    │    ✗†    │    ✗†    │     ✗†     │
└───────────────────┴──────────┴──────────┴──────────┴────────────┘

✓  = Full access
✓* = Access scoped to own faculty via FakultasScopeMiddleware
✗  = 403 Forbidden
✗† = Redirected away (already authenticated)
```

### 7.11.2 Faculty Scoping for Admin Routes

When an `admin_fakultas` user accesses `/admin/mahasiswa`, the `FakultasScopeMiddleware` modifies the underlying Eloquent queries to only return students whose `prodi.fakultas_id` matches the admin's own `fakultas_id`. The admin sees and can modify only students within their faculty.

Superadmins bypass this scoping entirely and see all records across all faculties.

### 7.11.3 No Cross-Role Data Sharing via Routes

There is no route that lets a lecturer view the admin dashboard, or lets an admin submit a KRS. Each role's routes are completely isolated. Shared functionality (profile management, notifications) lives in role-neutral routes at the root level.

The LMS, Skripsi, and Kerja Praktek modules appear under all three role prefixes (`/admin/lms`, `/dosen/lms`, `/mahasiswa/lms`), but each uses a different controller class with different permissions. The admin controller provides full CRUD, the dosen controller provides management actions, and the mahasiswa controller provides read-only access with submission capability.

## 7.12 No API Routes

SIAKAD does not define any routes in `routes/api.php`. The file either does not exist or is empty. All HTTP interactions follow the traditional server-rendered pattern:

```
Browser → GET /mahasiswa/krs → Server renders Blade view → Full HTML response
Browser → POST /mahasiswa/krs → Server processes form → Redirect with flash message
```

Three endpoints deviate from this pattern by returning non-HTML responses:

| Endpoint | Response Type | Reason |
|----------|--------------|--------|
| `GET /health` | JSON | Machine-readable status for monitoring tools |
| `POST /mahasiswa/ai-advisor/chat` | JSON | AJAX-driven chat interface requires structured data |
| `GET /mahasiswa/export/*` | PDF (binary) | Downloadable document generation |

Some admin controllers return partial HTML via AJAX for dynamic table filtering, but these are still Blade-rendered fragments — not JSON API responses.

This architecture means:
- No CORS configuration is needed.
- No API authentication tokens (Sanctum, Passport) are used.
- Every form submission uses CSRF tokens via Blade's `@csrf` directive.
- State management relies on server-side sessions, not client-side tokens.
- All validation errors are returned as redirect-back responses with flashed error bags, following Laravel's standard `$request->validate()` behavior.
